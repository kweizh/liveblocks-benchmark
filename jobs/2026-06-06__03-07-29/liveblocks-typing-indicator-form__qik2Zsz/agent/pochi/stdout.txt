# Multiplayer Form with Per-Field Typing Indicators and Field-Lock Overlay ## Background Liveblocks is a realtime collaboration infrastructure that exposes Presence (ephemeral per-user state) and Storage (CRDT-backed persistent state) through React hooks like `useMyPresence`, `useOthers`, `useStorage`, and `useMutation`. Your project is a Next.js (App Router) app that already renders a static form skeleton (Name / Email / Bio fields) but has no realtime behavior wired up. Your job is to turn it into a multiplayer form: every keystroke is synced across browsers via Liveblocks Storage, and every focused/typing user is visible to others via Liveblocks Presence, with a per-field "X is typing in Bio…" indicator and an optimistic per-field lock overlay. You MUST use the real Liveblocks cloud — DO NOT mock the Liveblocks SDK, DO NOT stub the WebSocket, and DO NOT swap in a fake provider. The runtime environment will provide a real `LIVEBLOCKS_SECRET_KEY` (server-side) and `NEXT_PUBLIC_LIVEBLOCKS_PUBLIC_KEY` (client-side, public key starting with `pk_`). ## Requirements * Implement a Liveblocks-backed multiplayer form using `@liveblocks/client` and `@liveblocks/react` (App Router compatible). * Storage: a single `LiveObject` named `form` holding three string fields: `name`, `email`, `bio`. Initialize them to empty strings if the room has no storage yet. * Presence: each user has a `typing` field whose value is one of `"name"`, `"email"`, `"bio"`, or `null`, plus an `info` (or `user`) field with a stable display name (e.g. derived from query-string `?name=Alice`, falling back to `Anonymous`). * Per-field input: each of the three `<input>`/`<textarea>` elements must be controlled by the value from Storage; every keystroke must write back into the `LiveObject` (e.g. with `useMutation`). The same field MUST stay in sync across all open browser tabs/contexts connected to the same room. * Per-field typing indicator: above each field, when at least one *other* user has `presence.typing` equal to that field id, render a visible indicator line whose text contains both the user's display name and the human-readable field label (Name / Email / Bio). Examples that are acceptable: `Alice is typing in Bio...`, `Alice is typing in Bio…`. The indicator MUST disappear when that other user blurs / stops typing in the field. * Optimistic per-field lock overlay: when at least one *other* user is currently `typing` in a field, the corresponding input for the local user MUST show a visually disabled overlay (greyed out / semi-transparent). The input itself MUST remain functionally usable — the local user can still focus it and take over typing (this is a soft lock, not a hard `disabled`). The overlay element MUST be detectable from the DOM (see Acceptance Criteria for the required marker). * The room id must be the same for every visitor of the page so two browser contexts collaborate in the same room. Hardcode the room id as `harbor-typing-form-${ZEALT_RUN_ID}` (read `ZEALT_RUN_ID` from the environment at build/start time; if unset, fall back to `harbor-typing-form-local`). This isolates each evaluation run on the real Liveblocks cloud. * The Liveblocks auth endpoint MUST be implemented on the server (e.g. `app/api/liveblocks-auth/route.ts`) using `LIVEBLOCKS_SECRET_KEY`.
