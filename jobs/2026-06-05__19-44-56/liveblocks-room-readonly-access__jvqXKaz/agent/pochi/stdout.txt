# Liveblocks Next.js: Role-Based Read-Only Room Access (Access Tokens) ## Background You are building a small collaborative dashboard powered by [Liveblocks](https://liveblocks.io) where each room has two kinds of participants: - **Editors** can view and mutate the shared state. - **Viewers** can only view the shared state in real time; they must **not** be able to mutate it. Liveblocks supports this directly via **Access Token authentication** (`liveblocks.prepareSession`) with `session.allow(roomId, session.READ_ACCESS)` or `session.allow(roomId, session.FULL_ACCESS)`. The server is responsible for inspecting the requested role and granting either read or full access to the room before authorizing the session. A partial Next.js 14 (App Router, TypeScript) project skeleton already exists at `/home/user/myproject`. All Node dependencies — including `@liveblocks/client`, `@liveblocks/react`, `@liveblocks/node`, and `next` — are pre-installed. The `/api/liveblocks-auth` route handler currently exists as a stub that returns HTTP 501 (Not Implemented); the page route exists as a non-functional placeholder. Your job is to make the system real. ## Requirements - Implement the `POST /api/liveblocks-auth` route handler at `app/api/liveblocks-auth/route.ts` using `@liveblocks/node` and the **Access Token** approach (`liveblocks.prepareSession` + `session.allow` + `session.authorize`). - The request body is JSON with at least the following fields: - `room` (string, required): the Liveblocks room id the client wants to enter. - `userId` (string, required): a mock-authenticated user id (the task is not building real auth, just trusting the request body for the role and identity). - `role` (string, required): one of `"editor"` or `"viewer"`. Any other value must be rejected. - Permission rules implemented through `session.allow(...)`: - When `role === "editor"`, grant `session.FULL_ACCESS` to the requested `room`. - When `role === "viewer"`, grant `session.READ_ACCESS` to the requested `room`. - On a valid request, call `session.authorize()` and return the upstream Liveblocks `{ status, body }` verbatim (status 200 with a JSON body containing a `token` string on success). - Reject malformed or invalid requests with `400 Bad Request` and a JSON body of the shape `{ "error": <string> }`.
