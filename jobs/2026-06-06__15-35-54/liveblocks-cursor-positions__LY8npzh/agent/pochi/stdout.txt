# Liveblocks Live Cursor Positions (Next.js) ## Background Build a small Next.js page that uses Liveblocks Presence to broadcast the current user's pointer position to everyone else in the same room, and render the cursors of all other connected users in realtime. This task exercises the core Liveblocks Presence primitives (`useMyPresence` / `useUpdateMyPresence` / `useOthers`) using the v2 React API. ## Requirements * Implement a single Next.js page at the route `/` that: * Wraps the page in a `LiveblocksProvider` (configured with the public API key from the environment) and a `RoomProvider` connected to the room whose id equals the value of `NEXT_PUBLIC_ZEALT_RUN_ID`. * Sets an initial Presence of `{ cursor: null }`. * On every `pointermove` over the page area, updates the local user's Presence with the cursor coordinates `{ x, y }` using `useUpdateMyPresence` (or `useMyPresence`). * On `pointerleave`, clears the local cursor by setting Presence to `{ cursor: null }`. * Renders one cursor element for every other connected user using `useOthers`. Each cursor element MUST: * Be absolutely positioned at the other user's `{ x, y }` Presence coordinates. * Have an HTML attribute `data-cursor` set to the user's connection id (e.g. `data-cursor="3"`). * Have a CSS class containing `cursor` (e.g. `class="cursor ..."`) so the verifier can locate it. * Be a colored marker (SVG or a colored div) — the color may be derived from the connection id. * The page MUST be visible at `http://localhost:3000/`. * Read the runtime room id from `process.env.NEXT_PUBLIC_ZEALT_RUN_ID` so the test harness can isolate concurrent runs. ## Implementation Hints * Use the v2 React API (`@liveblocks/client` + `@liveblocks/react`). Import hooks from `@liveblocks/react` and configure types in a `liveblocks.config.ts` file (use `type Presence = { cursor: { x: number, y: number } | null }`). * Authenticate from the browser using the public API key. With `LiveblocksProvider`, pass `publicApiKey={process.env.NEXT_PUBLIC_LIVEBLOCKS_PUBLIC_KEY!}`. * Frontend env vars in Next.js MUST be prefixed with `NEXT_PUBLIC_` so they are exposed to the browser. * Before starting the dev server, export the run id so the client picks it up: `export NEXT_PUBLIC_ZEALT_RUN_ID=$ZEALT_RUN_ID`. * Use a Next.js client component (`"use client"`) for the cursor page, since Liveblocks hooks run in the browser. * Use `useOthers((others) => others)` (or a mapped variant) to get the list of other users and render one cursor element per user. * After you have verified locally that the dev server starts and the page loads, **kill the dev server** before completing the task. The verifier will start the server itself. ## Acceptance Criteria * Project path: /home/user/myproject * Start command: `cd /home/user/myproject && NEXT_PUBLIC_ZEALT_RUN_ID=$ZEALT_RUN_ID NEXT_PUBLIC_LIVEBLOCKS_PUBLIC_KEY=$NEXT_PUBLIC_LIVEBLOCKS_PUBLIC_KEY npm run dev -- -p 3000` * Port: 3000 * The page at `http://localhost:3000/` loads without runtime errors and connects to the Liveblocks room whose id is the value of `$ZEALT_RUN_ID`. * When two browser sessions open the page simultaneously and one of them moves the pointer: * The other session sees at least one DOM element with class containing `cursor` and attribute `data-cursor` set to a connection id. * That element is positioned (via inline `style.left` / `style.top` or `transform`) at coordinates derived from the moved pointer. * The Liveblocks Active Users REST endpoint `GET https://api.liveblocks.io/v2/rooms/{roomId}/active_users` reports at least one active user while a browser is connected to the room (using `$ZEALT_RUN_ID` as room id). * The dev server MUST NOT be running when the task is marked complete (verifier starts its own instance).
