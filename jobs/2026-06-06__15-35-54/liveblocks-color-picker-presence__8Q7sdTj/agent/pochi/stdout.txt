# Liveblocks Color Picker Presence (Next.js + React) ## Background Build a small multiplayer Next.js page using the Liveblocks v2 React SDK. Each connected user picks one color from a fixed 5-color palette; the selected color is stored in their Presence and is visible to every other connected user in real time. The page also renders an avatars row showing each connected user with a background reflecting their currently selected color. ## Requirements * A single Next.js page at route `/` that joins a Liveblocks room scoped by the current `run-id`. * A 5-color palette grid containing exactly these colors (in this order): `red`, `blue`, `green`, `yellow`, `purple`. * Clicking a swatch updates the current user's Presence so the choice is broadcast to other users in the room. * An avatars row that shows one avatar per connected user (self + others), with each avatar visually reflecting that user's chosen color. * Each connected user has a stable `userId` taken from the `?userId=...` URL query parameter (e.g. `?userId=A`). * Real Liveblocks connectivity using the official Liveblocks REST/WebSocket service (no mocks, no in-memory fake provider). ## Implementation Hints * Use the Liveblocks v2 React SDK: `@liveblocks/client`, `@liveblocks/react`. * Use the `useUpdateMyPresence` hook to publish the locally chosen color, and `useOthers` (plus `useSelf` if needed) to render avatars for all connected users. * For client-side authentication, use the public Liveblocks key directly via `LiveblocksProvider` (`publicApiKey={...}`) — no auth endpoint is required for this task. * The Liveblocks public key must be exposed to the browser via a `NEXT_PUBLIC_`-prefixed environment variable. * The `run-id` is provided at runtime via the `ZEALT_RUN_ID` environment variable. Re-export it as `NEXT_PUBLIC_ZEALT_RUN_ID` (for example in `next.config.js` via the `env` field, or by reading `process.env.ZEALT_RUN_ID` at build/start time) so the browser can compute the same room id. * The room id used by `RoomProvider` MUST be exactly `color-picker-${run-id}`. * The user id read from `?userId=...` must be passed into `RoomProvider` as the room session via the `initialPresence` (store `userId` and `color` in Presence). The `userId` query parameter is required; if it is missing, the page may render an error or empty state. * Use plain client-side rendering (`"use client"`) for the page so query parameters and Liveblocks hooks work in the browser. ## Acceptance Criteria * Project path: /home/user/myproject * Start command: `npm run dev` (Next.js dev server) * Port: 3000 * Route: `/` (must accept a `?userId=<id>` query parameter) * Liveblocks room id: `color-picker-${run-id}` where `run-id` is read from the `ZEALT_RUN_ID` environment variable (re-exported as `NEXT_PUBLIC_ZEALT_RUN_ID` for the browser). * A `LIVEBLOCKS_PUBLIC_KEY` and a `NEXT_PUBLIC_LIVEBLOCKS_PUBLIC_KEY` (with the same value) must be readable from the environment; the frontend MUST authenticate using the `NEXT_PUBLIC_` variant via `LiveblocksProvider`'s `publicApiKey` prop. * Color palette: the page MUST render exactly 5 color swatches in this order: `red`, `blue`, `green`, `yellow`, `purple`.
