# Liveblocks Shared Pixel Grid ## Background Build a real-time collaborative "pixel grid" in a Next.js app using Liveblocks v2. The grid is a 10x10 board of clickable cells. Clicking a cell cycles its color through a fixed 4-color palette, and the color of each cell is persisted in a shared `LiveMap<string, string>` in Liveblocks Storage so every connected browser sees the same grid. ## Requirements * A Next.js page at `/` that renders exactly 100 clickable cells arranged in a 10-row by 10-column grid. * Each cell must expose a stable `data-cell` attribute of the form `"<row>-<col>"` where `row` and `col` are integers `0..9` (e.g. `data-cell="0-0"`, `data-cell="9-9"`). * Cell color state is stored in Liveblocks Storage under a top-level `LiveMap` named `pixels` whose keys are `"<row>-<col>"` strings and whose values are color names from the palette. * The palette is exactly the following four lowercase color names, in this cycle order: `red`, `blue`, `green`, `yellow`. * Clicking a cell sets its entry in `pixels` to the next color in the cycle, in this exact order: an unset cell becomes `red` on the first click, then `blue`, then `green`, then `yellow`, then wraps back to `red`. * All connected browsers must see updates in real time; do not write color state to any local file or local-only state. * The Liveblocks room is shared and isolated per evaluation run; the room ID must be derived from `ZEALT_RUN_ID` at runtime (see Implementation Hints). * Use Liveblocks v2 SDK and the Liveblocks v2 REST API. Real Liveblocks credentials are provided via environment variables — do not mock or stub Liveblocks. ## Implementation Hints * Use Next.js (App Router or Pages Router is fine) running on port `3000`. * Install and use the latest v2 packages: `@liveblocks/client`, `@liveblocks/react`, and `@liveblocks/node`. * Authenticate the frontend via a custom Next.js auth endpoint that uses `Liveblocks.prepareSession` (access tokens) from `@liveblocks/node`, granting `room:write` to the current room only. Initialize it via `initialStorage` so the map exists as soon as the first client connects (an empty `LiveMap` is fine). * Use `useMutation` (from `@liveblocks/react`) to read the current value of a cell from storage and write the next palette color, computing the next color from the cycle order. * Render the grid as plain DOM elements (e.g., `<button>` or `<div>`); the background color of each cell should visually reflect its current palette value but the verifier reads state from the Liveblocks REST API, not the DOM. * The verifier opens the page in headless Chromium, clicks cells using the `data-cell` selector, then calls the Liveblocks v2 REST endpoint `GET https://api.liveblocks.io/v2/rooms/{roomId}/storage?format=json` (with `Authorization: Bearer $LIVEBLOCKS_SECRET_KEY`) to read the final storage as JSON. ## Acceptance Criteria * Project path: /home/user/myproject * Start command: `cd /home/user/myproject && NEXT_PUBLIC_ZEALT_RUN_ID="$ZEALT_RUN_ID" npm run start` * Port: 3000 * Routes: * `GET /`: Renders the 10x10 pixel grid.
