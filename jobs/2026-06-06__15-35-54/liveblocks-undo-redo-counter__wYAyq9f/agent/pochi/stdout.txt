# Liveblocks Undo/Redo Counter (Next.js) ## Background Build a small Next.js page that demonstrates Liveblocks' built-in undo/redo history for collaborative Storage. The page renders a single numeric counter persisted in a Liveblocks room's Storage (`LiveObject { count: number }`), along with four buttons: **Increment**, **Decrement**, **Undo**, and **Redo**. Every increment or decrement must be a distinct entry on the Liveblocks history stack so that pressing **Undo** reverses one step at a time. This task uses **Liveblocks SDK v2** (`@liveblocks/client@^2`, `@liveblocks/react@^2`) and the **Liveblocks v2 REST API** (`https://api.liveblocks.io/v2/...`) for verification. ## Requirements * Implement a Next.js (App Router) page at `/` that joins a single Liveblocks room. * The room name **MUST** be derived from the `ZEALT_RUN_ID` environment variable so concurrent runs do not collide. * Storage root is a `LiveObject` containing a numeric `count` field. Initialize storage on the client when missing so that `count` starts at `0`. * The page must show: * The current value of `count` from Storage, inside an element with `data-testid="count"`. * A button with `data-testid="increment"` (labelled `Increment`). * A button with `data-testid="decrement"` (labelled `Decrement`). * A button with `data-testid="undo"` (labelled `Undo`). * A button with `data-testid="redo"` (labelled `Redo`). * Increment / decrement must mutate `count` via a Liveblocks mutation. Each click must produce a **separate** history entry so that one Undo call reverses exactly one click. * Undo / Redo must be wired to the `useUndo` and `useRedo` hooks from `@liveblocks/react`. * Use a public Liveblocks key (read from `NEXT_PUBLIC_LIVEBLOCKS_PUBLIC_KEY`) with `createClient({ publicApiKey })`; no custom auth endpoint is required. * Re-export the run id to the browser as `NEXT_PUBLIC_ZEALT_RUN_ID` so that the page can build the room name client-side. ## Implementation Hints * Liveblocks v2 React: wrap the page in `LiveblocksProvider` and a `RoomProvider`.
