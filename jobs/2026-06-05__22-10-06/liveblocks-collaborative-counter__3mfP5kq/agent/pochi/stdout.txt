# Liveblocks Collaborative Multi-Counter Dashboard ## Background You are extending a Next.js (App Router) starter application that already wires up a Liveblocks room. The room must host a collaborative multi-counter dashboard where every connected client sees the same set of named counters in real time. The starter has a placeholder page with no Storage-backed behavior yet ÔÇö your job is to make every counter creation, increment, decrement and reset propagate through Liveblocks Storage so that multiple browsers stay in sync. ## Requirements * The dashboard lives at the dynamic route `/dashboard/[room]` (App Router). Visiting `/dashboard/foo` joins a Liveblocks room named `foo`. * Storage layout (must match exactly): `Storage.counters: LiveMap<string, LiveObject<{ value: number; lastIncrementBy: string; lastUpdated: number }>>`. * A `RoomProvider` declared with `initialStorage={{ counters: new LiveMap() }}` so first-visit rooms get an empty map. * A user identity input (any text field) saved client-side; this value is the one written into `lastIncrementBy` on every Storage mutation. * A "Create counter" form with two inputs (counter key text + numeric initial value) and a submit button labelled `Create counter`. Submitting must add a new `LiveObject` entry to `Storage.counters` under the given key with the provided initial `value`, `lastIncrementBy = current identity`, and `lastUpdated = Date.now()`. If the key already exists in the map, do not overwrite it. * For every counter currently in the map, render a row containing: * the counter key, * the current value, * a `+` button labelled exactly `Increment` that adds 1 to `value`, * a `-` button labelled exactly `Decrement` that subtracts 1 from `value`, * a text line of the form `Last touched by <identity> at <ISO timestamp>` (use `new Date(lastUpdated).toISOString()`). * Every increment/decrement must update `value`, `lastIncrementBy`, and `lastUpdated` together inside a single `useMutation` callback (Liveblocks automatically batches updates inside `useMutation`). * A global `Reset all` button that, inside a single `useMutation`, iterates every counter in `Storage.counters` and sets each one's `value` back to `0`, `lastIncrementBy = current identity`, and `lastUpdated = Date.now()` so that other clients receive the reset as one batched update. * Use **real** Liveblocks cloud ÔÇö never mock the client, never stub Storage, never fake the WebSocket. Authentication must go through a server route at `/api/liveblocks-auth` that uses `LIVEBLOCKS_SECRET_KEY` to mint a session via `@liveblocks/node`. The browser must access the room via the official `@liveblocks/react` provider configured to use that auth endpoint. `NEXT_PUBLIC_LIVEBLOCKS_PUBLIC_KEY` may be used as a fallback `publicApiKey`, but the auth endpoint route MUST be present and functional. ## Implementation Hints * Read the room name from the route segment `[room]` and pass it down as the `RoomProvider id`. * Type Storage globally so `useStorage((root) => root.counters)` returns a typed `ReadonlyMap`.
