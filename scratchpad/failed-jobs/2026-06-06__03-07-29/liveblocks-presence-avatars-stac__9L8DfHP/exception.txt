# Liveblocks Presence — Avatar Stack Header

## Background
You are extending an existing Next.js scaffold that already includes a `LiveblocksProvider`, a `RoomProvider`, an auth endpoint, and a placeholder `Header` component. The scaffold connects to the **real Liveblocks cloud** using the project's `LIVEBLOCKS_SECRET_KEY` and `NEXT_PUBLIC_LIVEBLOCKS_PUBLIC_KEY`.

Your job is to finish implementing a presence-driven **avatar stack** in the header that mirrors what teams expect from collaborative apps like Figma, Linear, or Notion: a horizontal row of overlapping user avatars showing who is currently in the room, with a `+N` overflow badge and per-user hover tooltips.

## Requirements
- Render a horizontal stack of overlapping circular avatars in the `Header` for every user currently connected to the room (including the current user).
- The current user (from `useSelf()`) must be shown as the **leftmost** avatar and visually marked as "You" in the tooltip.
- Other connected users (from `useOthers()`) must be rendered to the right of the current user, ordered by their `connectionId` ascending so the order is stable across all clients.
- Show **at most 4 avatars** in the stack. If there are more connected users, render a single trailing **`+N` overflow badge** where `N` is the number of hidden users (total connected users minus 4).
- Each avatar must:
  - Use the user's `info.avatar` URL as the image source if present, with the user's `info.name` initials as a fallback when the image fails to load.
  - Have its border / outline coloured with the user's `info.color` value.
  - Show a small **green online dot** in the bottom-right corner of the avatar.
  - Reveal a **tooltip on hover** that displays the user's `info.name`. For the current user the tooltip text must be of the form `<name> (You)`.
- The user info (`name`, `avatar`, `color`) must be supplied to the client via the existing auth route by attaching a `userInfo` object inside `identifyUser` on the server side. Do not hard-code user info on the client.
- The header must continue to render correctly when there is only the current user in the room (stack of 1, no `+N` badge) and when the room is empty during initial connection (placeholder/skeleton, no crash).
- Do **NOT** mock Liveblocks. The implementation must talk to real Liveblocks cloud using the env vars provided in the environment.

## Implementation Hints
- The relevant React hooks are `useSelf` and `useOthers` from `@liveblocks/react/suspense`.
- `useOthers()` returns an immutable array of `User` objects, each exposing `connectionId`, `presence`, and `info`.
- On the server, `Liveblocks.identifyUser({ userId, groupIds }, { userInfo })` from `@liveblocks/node` is how you attach static metadata (name, avatar, color) to a session so it becomes available on `user.info` on the client.
- The scaffold provides a list of fixed mock users keyed by a `?user=<index>` query parameter on the page route, so you can open multiple browser contexts with different users to simulate presence. Use that mechanism — do not invent a new auth flow.
- For the tooltip, any pure-CSS hover tooltip is sufficient. You do not need a tooltip library.
- For the `+N` badge, treat it as a single DOM node that sits at the right end of the stack.
